Implementation Approaches

Three ways to build MapMyTrain — pick your starting point

A

Monorepo Full Stack

Single repository with all services. Docker Compose orchestrates everything. Fastest to iterate, simplest to debug.

Structure
map-my-train/
├── frontend/ (Next.js 14)
├── backend/ (FastAPI + Celery)
├── ingestion/ (Scraper workers)
├── database/ (PostGIS schemas)
├── docker-compose.yml
└── .env.example

Pros

  • One git repo, one deploy
  • Shared types/interfaces
  • Easiest local dev setup

Cons

  • Harder to split later
  • Public/private split needs work
B

Dual Repository (as designed)

Follow the manifesto exactly. Public core repo + private prod repo with submodule. Production-ready from day one.

Structure
map-my-train-core/ (public)
├── frontend/
├── base-backend/
└── database/
map-my-train-prod/ (private)
├── core/ ← submodule
├── premium/
└── .env.production

Pros

  • Matches architecture docs
  • Clean IP separation
  • Community can contribute safely

Cons

  • More complex setup
  • Submodule sync overhead
C

Phased Monorepo → Split

Start monorepo for speed, extract to dual-repo once core is stable. Best of both worlds with migration cost.

Phases
Phase 1: Monorepo (weeks 1-4)
└── Build everything in one repo
Phase 2: Extract (weeks 5-6)
└── Split public core, setup submodule
Phase 3: Production (week 7+)
└── Deploy with full dual-repo setup

Pros

  • Fastest initial development
  • Validate architecture first
  • Split when you need to

Cons

  • Migration work later
  • Temp IP overlap risk
